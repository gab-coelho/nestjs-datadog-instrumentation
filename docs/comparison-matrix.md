# Matriz de comparação

Preencha esta matriz com as observações reais do laboratório: contagem de traces, nomes de spans, spans ausentes, diferenças de latência e qualquer comportamento inesperado.

| Dimensão | Single Step Instrumentation (SSI) | SDK manual `dd-trace` |
|---|---|---|
| **Complexidade de preparação** | Baixa. Não exige mudança de código nem de `package.json`. A configuração fica no Agent/Admission Controller e a ativação acontece via annotations nos pods ou labels nos namespaces. | Moderada. Exige adicionar a dependência `dd-trace`, manter a configuração de `tracer.init()` e garantir que o tracer carregue antes dos demais módulos, que é uma fonte comum de bugs. |
| **Mudanças no deploy Kubernetes** | Em geral, apenas annotations ou labels. Neste lab, `ssi-instrumentation/k8s/deployment.yaml` usa a imagem base e ativa SSI por annotation no pod. O admission webhook injeta init container, variáveis de ambiente e volumes em runtime; isso reduz mudanças no YAML, mas deixa parte do comportamento menos visível no controle de versão. | O YAML de deployment carrega variáveis explícitas como `DD_AGENT_HOST`, `DD_ENV`, `DD_SERVICE` e `DD_VERSION`. Não depende de um admission webhook funcionando corretamente no cluster inteiro. |
| **Configuração** | Centralizada no agent, com boa consistência entre muitos serviços. Personalizações por serviço, como sampling rules, tags customizadas e spans manuais, tendem a exigir SDK manual de qualquer forma. | Totalmente customizável: spans, tags, propagação de contexto e ajustes finos de integrações. O custo é cada serviço passar a possuir essa configuração. |
| **Manutenção** | Upgrades podem acontecer centralmente, atualizando a versão injetada pelo Agent. É fácil distribuir para muitos serviços, mas é preciso inspecionar pods reais para saber exatamente o que foi injetado. | A versão do `dd-trace` é uma dependência explícita de cada serviço. Isso gera mais PRs e deploys, mas deixa as mudanças rastreáveis no histórico do repositório. |
| **Cobertura de observabilidade** | Cobre bem bibliotecas suportadas por auto-instrumentation, como HTTP, frameworks comuns e alguns clients de banco. Não cobre spans de lógica de negócio. | Permite cobertura mais profunda, incluindo spans de negócio, tags manuais e propagação customizada. O teto é maior, mas depende de escrever e manter essa instrumentação. |
| **Experiência de desenvolvimento** | Baixo esforço para desenvolvedores; tracing fica mais como responsabilidade de plataforma/infraestrutura. Depurar spans ausentes pode ser menos óbvio, porque a instrumentação não está no código da aplicação. | Desenvolvedores veem e controlam o código de tracing. Fica mais fácil raciocinar localmente, mas há mais dependência, configuração e superfície operacional em cada serviço. |
| **Falha silenciosa esperada** | Se a annotation/label de admission faltar ou o webhook não atuar, o pod pode subir sem instrumentação. Deve ser validado por inspeção do pod e pelo APM. | Se o preload do tracer falhar ou for removido, a aplicação pode subir normalmente sem traces. Deve ser validado com uma quebra proposital no laboratório. |
